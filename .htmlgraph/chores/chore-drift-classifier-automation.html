Problem

HtmlGraph detects drift in activity tracking (when activities don't align with the current feature), but currently:

  • Drift detection happens post-hoc through analysis
  • No automated classification of drift patterns into work items
  • Users must manually create issues for drift-related work
  • Drift queue accumulates without actionable items

Solution

Create automation to classify high-drift activities into appropriate work items:

  1. Monitor drift-queue.json for activities with drift_score > 0.7
  2. Analyze activity patterns to determine work item type (bug, feature, spike, chore, hotfix)
  3. Generate work item HTML file with appropriate metadata
  4. Link work item back to the high-drift activities that triggered it
  5. Clear processed activities from drift queue

Classification Criteria

Bug
Errors, incorrect behavior, exceptions in drift detection or tracking
Feature
New functionality for drift handling, reporting, or visualization
Spike
Research, exploration, investigation of drift patterns
Chore
Maintenance, cleanup, infrastructure for drift management
Hotfix
Urgent issues blocking feature progress (rare)

Acceptance Criteria

  • Drift activities with score > 0.7 are automatically classified into work items
  • Work items are created with appropriate type, title, and description
  • Work items link back to triggering activities in drift queue
  • Processed drift activities are tracked/cleared from queue
  • Classification rules are testable and documented
  • CLI command available: htmlgraph drift classify

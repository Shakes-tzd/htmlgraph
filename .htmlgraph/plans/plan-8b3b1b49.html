A. Design DiscussionPending

Problem

Development teams using AI agents (Claude Code, Cursor, Copilot) jump straight from idea to code. There is no structured "plan review" step where humans approve scope, answer design questions, and see AI critique — before a single line is written.

HtmlGraph built this internally as the CRISPI plan system (Context, Request, Intelligence, Slices, Plan, Implementation). It works — but it is locked inside a Go binary with HtmlGraph-specific assumptions. Extracting it as a standalone Python tool ("planbook") lets any team adopt plan-first development.

Unique differentiators: (1) AI critique of plans pre-implementation, (2) structured feedback persisted to SQLite, (3) reactive notebook-based interactive review.

Target Audience (MVP)

Solo developers and small teams (2-5) using AI coding agents. Plans become the "steering spec" that agents execute against. The 85k stars on spec-kit prove demand for structured plan-before-code workflows.

Packaging Decision

Python package distributed via PyPI: pip install planbook (core) or pip install planbook[review] (includes marimo). CLI commands: planbook init, planbook validate, planbook critique, planbook review, planbook export. Also importable as a library for agent framework integration.

Open Questions

Should planbook be a Python package, CLI tool, or both?

Both gives maximum flexibility — solo devs use CLI, agent frameworks import the library. cli-only is simpler but limits programmatic use.

Should marimo be a required or optional dependency?

Optional keeps the core package lightweight. Required simplifies install but forces marimo on users who only want validate+export. Peer dependency is confusing for new users.

How should the YAML schema be validated?

Pydantic gives the best developer experience (autocomplete, error messages, Python types). JSON Schema is more portable but less ergonomic. Flexible risks silent errors.

Who is the primary target user for MVP?

Solo devs are the easiest first market — no auth/multi-user needed. spec-kit's 85k stars prove demand. Teams need collaboration features (comments, assignments). Agent frameworks need stable API.

What should the MVP scope include?

yaml-plus-review ships the core review loop without the complexity of LLM integration. Full is ambitious but risks delayed launch. yaml-only proves nothing — the review loop IS the product.

Key Design Choices

  • Pydantic for YAML schema — strict validation with clear error messages, JSON Schema export for editor autocomplete
  • Marimo as optional dependency — planbook[review] extra; core works without it (validate, critique, export all CLI-only)
  • SQLite for feedback — zero-config, file-based, queries for approval state; one DB per project
  • LiteLLM for AI critique — provider-agnostic (OpenAI, Anthropic, local); no vendor lock-in
  • Jinja2 for HTML export — self-contained HTML5 with embedded CSS/JS, no CDN calls

MVP Scope (v0.1)

YAML schema + validation + marimo review + feedback persistence + HTML export. AI critique deferred to v0.2 (it is the biggest differentiator but also the most complex — shipping review-first proves the core value).

What is NOT in scope

  • Authentication / multi-user (local-first, single user)
  • Cloud sync (git is the sync mechanism)
  • VS Code extension (CLI + notebook is sufficient for MVP)
  • Plan versioning / diffing (git handles this)
  • Integration with Linear/Jira/GitHub Issues (future)

#1YAML Schema & CLI Skeleton M Low RiskPending
Files: src/planbook/schema.py, src/planbook/cli.py, src/planbook/__init__.py, templates/plan.yaml

Pydantic models for CRISPI plan YAML (Plan, Slice, Question, Critique sections). Click CLI with init/validate commands. planbook init creates starter plan.yaml from template. Foundation for everything — all other slices consume the validated Plan model.

Test Strategy

  • Unit: Plan model validates valid YAML, rejects missing title, rejects unknown fields
  • Unit: planbook init creates valid YAML that passes validate
  • Unit: JSON Schema export matches Pydantic model
  • Integration: CLI init + validate roundtrip

Dependencies: none

#2SQLite Feedback Store S Low RiskPending
Files: src/planbook/store.py

Feedback persistence layer: create/read/update feedback records keyed by (plan_id, section, action). IsPlanFullyApproved() query. One .planbook.db per project. Auto-created on first write.

Test Strategy

  • Unit: persist_feedback() upserts correctly (insert then update same key)
  • Unit: get_feedback() returns structured dict
  • Unit: is_fully_approved() returns false when any section missing
  • Integration: DB auto-created on first write, survives process restart

Dependencies: none

#5AI Critique Engine (v0.2) L High RiskPending

Dual-agent critique: two LLMs independently review, synthesis merges findings. LiteLLM for provider flexibility. planbook critique writes assumption verification + risk table + synthesis to YAML critique section. Complexity-gated: skip for <3 slices. Deferred to v0.2.

Test Strategy

  • Unit: complexity gate skips for <3 slices

Plan: Plan-First Architecture Redesign

Redesign plan commands so plans are the starting point, not a review layer on existing tracks. Plan-first creates, plan finalize generates the work item graph.

Finalized Track: trk-bbf223f6 (CRISPI Plan Mode) | Generated: trk-f8fd5c66

Dependency Graph

A. Design DiscussionApproved

Goals

  1. Plans created from topic/title — no pre-existing track required
  2. plan finalize is generative — creates track, features, dependency edges, bakes question answers into descriptions
  3. Track-to-plan path preserved as "retroactive planning"
  4. Multi-AI critique integrated with clearer synthesis protocol
  5. Plans become first-class work items with hex8 IDs

Constraints

  • HTML canonical, SQLite read cache
  • No external infrastructure
  • Plan interactive review UI preserved
  • Existing plan commands must continue working

Assumptions

  • A1: Current plan template can extend for plan-first mode (no separate template needed)
  • A2: plan finalize can reuse workitem.Create() for track+feature generation
  • A3: Plan skill (plugin/skills/plan/SKILL.md) will be primary consumer of new CLI commands
  • A4: Plan IDs can migrate to hex8 without breaking existing plans

These assumptions were verified by AI critique — see the Critique Results section below.

Gemini — Design Review

Slice-by-Slice Assessment

  • S1 Well-scoped. Aligns with established feat-/bug-/trk- convention. Backward compat via prefix matching is safe.
  • S2 Focused. Risk: Orphaned draft plans that never finalize — consider garbage collection.
  • S3 Most complex slice. Idempotency risk: if finalize runs twice, must not recreate track or duplicate features. Needs rollback-on-error strategy. Missing internal/models/node.go from file list.
  • S4 Title collision: plan generate "trk-test" could be misinterpreted as track ID. Use regex/prefix checking.
  • S5 Should include --dry-run flag to estimate cost before firing dual LLM calls.
  • S6 Desync risk: skill must be updated in tandem with binary release.
  • S7 Use internal/htmlparse for structured assertions, not raw string matching.

Overall

  • Dependency ordering: Correct. S1→S2 must land first.
  • Biggest risk: S3 generative finalize — inconsistent state if creation fails mid-way.
  • Suggestion: Merge S1+S2 (highly coupled). Split S3 into S3a (track generation) and S3b (feature + edge generation).

Copilot — Implementation Exploration

What Exists to Leverage

  • Ready generateID("plan", title) at internal/workitem/htmlwriter.go:42 already produces hex8 IDs. S1 is smaller than estimated.
  • Ready workitem.Create() pattern at features.go:74, tracks.go, plans.go:53 — all take title + functional options.
  • Ready Plan feedback storage (internal/db/plan_feedback.go) fully implemented with tests.

What the Plan Gets Wrong

  • Divergence CLI uses derivePlanID() (slugs), library uses generateID() (hex8). This is a UNIFICATION need, not a migration.
  • Coupling Plan generation tightly coupled to pre-existing tracks — reads contains edges. Cannot work without a track.
  • One-shot Template system designed for one-time generation. set-slice assumes slices already exist — cannot CREATE new slices.

Missing Constraints

  • Breaking Plan skill orchestration assumes track-first. S6 is a REWRITE, not an update.
  • Gap Current finalize only sets status. Generative work is ~90% new code.
  • Migration Existing plan files mix slug and hex8 IDs. No migration strategy defined.

Simpler Alternatives

  • Option A: Two-phase creation — plan create (empty) → plan append-slice → finalize stays simple.
  • Option B: Plans ARE tracks with a special template — reuses existing graph.
  • Option C: Finalize writes manifest JSON → separate plan materialize creates work items.
  • Recommendation: Start with spike — create empty plan, verify open/wait work, build incrementally.

Synthesis

Consensus Findings

  • S1 is smaller than estimated. Both critics confirm generateID() already exists — the work is removing derivePlanID() and unifying.
  • S3 is the critical path and highest risk. Gemini flags idempotency and rollback. Copilot flags it is 90% new code. Consider splitting into S3a (track creation) and S3b (feature + edge generation).
  • S6 is a rewrite, not an update. Both critics agree the skill flow must completely reverse.
  • Template set-slice cannot create new slices — only update existing ones. Need append-slice or template rework.

Disagreements

  • S1+S2 coupling: Gemini suggests merging. Copilot treats as distinct. Recommendation: keep separate — S1 is small enough to land independently.

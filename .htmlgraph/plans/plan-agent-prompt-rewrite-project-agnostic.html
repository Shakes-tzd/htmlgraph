A. Design DiscussionPending

Current State

12 agent system prompts in plugin/agents/ share approximately 47 lines of identical boilerplate across all files (registration block + safety rules + development principles). The boilerplate comprises ~20-25% of smaller agents (codex/copilot/gemini operators at 86-94 lines) and ~15-20% of larger agents (debugger/roborev/test-runner at 213-258 lines). Additionally, 6 agents contain HtmlGraph codebase-specific references that break agent portability:

  • test-runner.md (258 lines): 8 hardcoded Go test commands with references to `packages/go/`, `go vet`, `go build`, GoReleaser. Lines 107-130.
  • roborev.md (213 lines): 8 HtmlGraph code review criteria (DRY, SRP) + Go-specific commands (`go build` errors, `go vet` warnings). Lines 58-81.
  • debugger.md (241 lines): HtmlGraph debugging scenarios (hook merging, orchestrator status), `.claude/settings.json`, hook environment variables. Lines 150-206.
  • researcher.md (195 lines): References to `htmlgraph find`, HtmlGraph database queries, spikes. Lines 89-105.
  • task-executor.md (89 lines): References to `.htmlgraph/` and `htmlgraph` commands. Lines 6-20.
  • gemini-operator.md (94 lines): Project-agnostic but includes HtmlGraph context guidance (gemini CLI usage pattern).

Impact: These agents fail or lose effectiveness when used in non-Go projects (Python/Ruby/JS). A Python developer using haiku-coder, sonnet-coder, or opus-coder gets HtmlGraph-specific guidance. A debugger used for JavaScript debugging gets Go test error examples.

Line count analysis: Total 1,858 lines. Average 155 lines/agent. Boilerplate overhead: ~270 lines (15% of total). After removing boilerplate and HtmlGraph-specifics, agents would be 1,200-1,300 lines — well within production guidelines (~1,700 words = ~340-400 lines at current density).

Desired End State

  • All 12 agents work in ANY project language (Go, Python, JavaScript, Ruby, Rust) while still enforcing HtmlGraph work tracking via CLI.
  • Shared boilerplate tightened to ~15-20 lines (down from 47). Consolidate to: one STOP block statement, one reference to .htmlgraph safety rules with CLI link, one sentence on dev principles.
  • Project-specific references (Go paths, internal packages, HtmlGraph-specific debugging scenarios) replaced with generic equivalents or moved to separate HtmlGraph-customization layer.
  • Each agent follows official Claude Code agent guidelines: second person, imperative language, clear when-to-use, ~1700 words (achieved via tighter boilerplate + focused content).
  • Coder agents: Remove HtmlGraph context guidance from frontmatter (triggerPatterns, when_to_use). These fields are silently ignored by Claude Code and add confusion.
  • Operator agents: Update model references to reflect current best practices (haiku for gemini-operator is underutilized; gemini/codex are research tools, not deployment/delegation tools).

Relevant Patterns & Constraints

  • Official agent format: YAML frontmatter (required fields: name, description, model, color, tools) + markdown body = system prompt.
  • Claude Code agent guidelines: Use second person ("You are..."), imperative voice, identity sentence, clear boundaries, output expectations.
  • Frontmatter constraint: Non-standard fields (triggerPatterns, when_to_use, when_not_to_use) are silently ignored by Claude Code; they serve no purpose in plugin agents and should be removed.
  • Skills injection limitation: Skills cannot auto-inject via frontmatter; boilerplate must stay inline in agent markdown.
  • Reusability requirement: Agents must work in any project. HtmlGraph-specific guidance should be optional context, not core responsibility.
  • Boilerplate persistence: Work registration, .htmlgraph safety, and development principles MUST stay in every agent (non-negotiable for HtmlGraph CLI integration).

Risk Assessment

High Risk (likely to lose effectiveness):

  • test-runner: Currently tightly coupled to Go. Needs smart auto-detection (look for go.mod, package.json, pyproject.toml) or generic "run your project's tests" guidance. Risk: Agent becomes too generic and loses practical value.
  • roborev: Currently enforces HtmlGraph design principles (DRY, SRP) + Go-specific quality gates. Genericizing to "follow your project's conventions" may dilute its purpose. Risk: Agent becomes advisory rather than enforceable.
  • debugger: HtmlGraph-specific scenarios (hook merging, orchestrator) are valuable examples. Removing them makes agent less concrete. Risk: Agent becomes abstract guidance without grounding.

Low Risk (straightforward genericization):

  • researcher, task-executor, coder agents: Already largely project-agnostic. Main work is removing `.htmlgraph` references and hardcoded command examples.
  • operator agents: Mostly boilerplate removal. Model/tool changes are organizational, not functional.

Open Questions

How aggressive should registration enforcement be after genericization?

The current STOP block (all caps, 'NOT ALLOWED', 'bug in your behavior') is maximally enforceable for HtmlGraph but sounds hostile in other projects. However user memory confirms registration should be mandatory. Balance compliance with tone appropriateness.

How should test-runner detect and run tests in non-Go projects?

test-runner currently has 8 hardcoded Go test commands (go test, go build, go vet patterns). Generic replacement must work for Python (pytest/unittest), Node (jest/mocha), Ruby (rspec), Rust (cargo test). Auto-detection is most portable; generic is simplest but least helpful.

What should roborev review criteria become after genericization?

roborev currently enforces HtmlGraph's DRY/SRP/KISS/YAGNI principles. These are good general principles but project-specific. Ruby project might prioritize different patterns. Hybrid approach provides guidance while respecting project norms.

Should debugger keep HtmlGraph-specific debugging scenarios as examples?

debugger.md includes concrete scenarios about hook merging and orchestrator enforcement (lines 178-206). These are valuable examples that ground abstract methodology. Removing them makes agent less concrete. Separate section allows flexibility — users ignore if irrelevant.

Should non-standard frontmatter fields be removed from coder agents?

haiku-coder sonnet-coder opus-coder use triggerPatterns and when_to_use frontmatter fields that Claude Code silently ignores. These fields add 20+ lines per agent with zero effect. Simpler to remove from boilerplate and keep only in body text where needed.

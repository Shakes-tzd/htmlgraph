The root cause is a two-layer problem: (1) The SQLite features table still shows 'in-progress' because reindex was run BEFORE the HTML was updated to 'done', and then no subsequent reindex was run to sync. (2) However, recommend.go does NOT use SQLite at all -- it calls graph.LoadAll(dir) which reads HTML files directly via htmlparse.ParseFile. So recommend's WIP count reads status='done' correctly from HTML. The inflated WIP count the bug reporter sees in 'htmlgraph recommend' is therefore NOT caused by stale SQLite. The actual inflation source is that the WIP count in recommend.go collectWIPItems() includes ALL nodes where n.Status==StatusInProgress, which reads from HTML -- but the spike HTML says 'done', so it should NOT appear. The actual bug is likely that htmlgraph wip or wip hooks read from SQLite (features table) rather than HTML, and the SQLite record is stale at 'in-progress'. Fix: run 'htmlgraph reindex' to sync the features table from HTML. Permanent fix: reindex should be idempotent and run automatically, or the bug.start / spike.start commands should ensure stale SQLite entries are corrected on next reindex.

## Problem Investigate whether subagent tool call nesting is correctly implemented in HtmlGraph - specifically whether Claude Code passes agentId/parentToolUseId for subagent calls, whether these are stored in the database, and whether the dashboard renders nesting. ## Research Sources ### 1. Official Claude Code Hooks Documentation (code.claude.com/docs/en/hooks) CONFIRMED fields in PreToolUse/PostToolUse hook input: - session_id, transcript_path, cwd, permission_mode, hook_event_name (common to all) - tool_name, tool_input, tool_use_id (PreToolUse/PostToolUse specific) - agent_id: Present ONLY when hook fires inside a subagent (unique subagent identifier) - agent_type: Present when session uses --agent or hook fires inside a subagent NO parentToolUseId field exists in the official schema. NO isSidechain field in hook input (only in transcript .jsonl files via transcript.py). SubagentStart hook input: session_id (PARENT session), agent_id (unique subagent id), agent_type SubagentStop hook input: session_id (PARENT session), agent_id, agent_type, agent_transcript_path, last_assistant_message KEY INSIGHT from docs: "When running with --agent or inside a subagent, two additional fields are included: agent_id and agent_type". So subagent tool calls DO carry agent_id. ### 2. HtmlGraph Hook Code Analysis #### PreToolUse (pretooluse.py) - create_start_event() - Reads agent_id from tool_input: native_agent_id = tool_input.get("agent_id") - If agent_id present: queries DB for task_delegation where agent_id = native_agent_id - If agent_id absent: falls back to session_id suffix-stripping heuristics - Creates task_delegation event in agent_events table for Task() calls - Inserts into tool_traces table (parent_tool_use_id set to None, never populated) #### PostToolUse (event_tracker.py) - track_event() - Parent linking priority order: 1. HTMLGRAPH_PARENT_EVENT_FOR_POST env var (set by PreToolUse same-process) 2. HTMLGRAPH_PARENT_EVENT env var (cross-process Task delegation) 3. HTMLGRAPH_PARENT_QUERY_EVENT env var (legacy fallback) 4. task_event_id_from_db (DB query for active task_delegation) 5. Active task_delegation in parent session (DB fallback with suffix-stripping) 6. UserQuery event (final fallback) #### SubagentStart (subagent_start.py) - Receives agent_id and agent_type from Claude Code - Finds earliest unmatched task_delegation (status=started, agent_id IS NULL) - First tries to match by agent_type == subagent_type - Falls back to first unmatched if no type match - Updates task_delegation.agent_id = agent_id (FIFO mapping) #### SubagentStop (subagent_stop.py) - Reads HTMLGRAPH_PARENT_EVENT env var or queries DB for most recent task_delegation - Counts child spikes created after parent event start time - Updates parent event: status=completed, child_spike_count=N, output_summary=JSON - NOTE: The old comment "SubagentStop cannot identify which specific subagent finished" is now outdated - SubagentStop DOES receive agent_id per official docs ### 3. Database Schema Analysis agent_events table has these hierarchy columns: - parent_event_id TEXT (FK to agent_events.event_id) - primary nesting link - parent_agent_id TEXT - agent delegation chain - subagent_type TEXT - type of spawned agent - agent_id TEXT - normalized agent identifier (used by SubagentStart mapping) - status TEXT - task_delegation lifecycle (started/completed) sessions table has: - parent_session_id TEXT (FK to sessions.session_id) - parent_event_id TEXT (FK to agent_events.event_id) - is_subagent BOOLEAN tool_traces table has: - parent_tool_use_id TEXT (FK to tool_traces.tool_use_id) - ALWAYS NULL (never set) ### 4. Dashboard Rendering Analysis activity-feed-hierarchical.html template: - DOES support parent/child rendering - Groups events as {parent, children, has_children} - Children are hidden by default, expandable via toggleChildren() - Uses data-parent attribute to link child rows to parent API route /views/activity-feed: - Uses activity_service.get_grouped_events() - Returns conversation_turns (grouped by UserQuery) - Renders activity-feed.html (NOT activity-feed-hierarchical.html) - The hierarchical template exists but is NOT the default view ## Root Cause Analysis ### Q1: Does Claude Code pass agentId for subagent tool calls? YES - confirmed by official docs. When a tool call fires inside a subagent, hook input includes agent_id (unique subagent identifier) and agent_type. ### Q2: Is parentToolUseId passed? NO - this field does not exist in Claude Code hook input. There is no parentToolUseId in PreToolUse or PostToolUse hook schema. ### Q3: Are agent_id and hierarchy stored in the database? PARTIALLY - Yes for task_delegation events (agent_id mapped by SubagentStart). PROBLEM: The tool_traces table has parent_tool_use_id column but it is ALWAYS NULL - the value is never populated in create_start_event(). The comment says "Will be set by SubagentStop hook" but SubagentStop never sets it. For agent_events: parent_event_id IS being set for tool calls inside subagents, using the multi-layered fallback system in PostToolUse handler. This works when: - HTMLGRAPH_PARENT_EVENT env var is set (same-process), OR - active task_delegation exists in DB with status=started ### Q4: Is the dashboard rendering hierarchy? PARTIALLY - The hierarchical template (activity-feed-hierarchical.html) exists and correctly renders parent/child rows with expand/collapse. HOWEVER: - The default /views/activity-feed route renders activity-feed.html (NOT hierarchical) - The hierarchical view appears to be a separate route (event-traces.html template) - The redesign dashboard (dashboard-redesign.html) groups by conversation turn but does not render tool-call-level nesting inside subagents ### Q5: What is the root cause of flat (non-nested) display? THREE ROOT CAUSES: 1. WRONG DEFAULT TEMPLATE: /views/activity-feed uses activity-feed.html which renders flat. The hierarchical template exists but is not the default view. 2. tool_traces.parent_tool_use_id NEVER SET: The column exists in schema and tool_traces records are created, but parent_tool_use_id is hardcoded to None in create_start_event() with the comment "Will be set by SubagentStop hook" - but SubagentStop never populates it. 3. ENVIRONMENT VARIABLE INHERITANCE LIMITATION: PreToolUse sets HTMLGRAPH_PARENT_EVENT in the parent process environment, but this may NOT be inherited by subagent tool call processes due to process boundary. SubagentStop comments note: "environment variables may not be inherited across subagent process boundary". This means the DB fallback (active task_delegation query) is the primary mechanism for linking. 4. SUBAGENT STOP MISSES agent_id: The SubagentStop handler does NOT use the agent_id field that Claude Code provides in the hook input. Instead it uses env var or "most recent task_delegation" heuristic. With parallel subagents, this can attribute the stop to the wrong task_delegation. ## Key Working Components - SubagentStart correctly maps agent_id -> task_delegation (FIFO with type matching) - PreToolUse correctly creates task_delegation events for Task() calls - parent_event_id IS being stored in agent_events for subagent tool calls - The hierarchical template (activity-feed-hierarchical.html) is ready to render the nesting if fed correct data ## Recommended Fix Areas 1. Wire /views/activity-feed to use activity-feed-hierarchical.html with proper hierarchical query (GROUP BY parent_event_id) 2. Fix SubagentStop to use agent_id from hook input to find the CORRECT task_delegation event (rather than "most recent"): SELECT event_id FROM agent_events WHERE agent_id = ? AND event_type = task_delegation 3. Fix tool_traces.parent_tool_use_id to be populated using agent_id lookup (find the parent task_delegation tool_use_id when inside a subagent) 4. The DB-based parent lookup in PostToolUse is the right architecture given env var inheritance limitations - it just needs the agent_id field to be used for more precise matching instead of session-level heuristics

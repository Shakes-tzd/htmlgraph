## Summary The activity feed shows a flat list instead of nested subagent tool calls. There are THREE separate breakdowns in the pipeline, each compounding the others. --- ## Evidence Gathered ### 1. Database: parent_tool_use_id is NULL for all rows in tool_traces Query: SELECT tool_name, parent_tool_use_id FROM tool_traces ORDER BY start_time DESC LIMIT 20 Result: ALL rows have parent_tool_use_id = None (including Bash, Agent, Read, etc.) ### 2. Database: agent_events parent_event_id is partially populated Query: SELECT tool_name, parent_event_id, agent_id FROM agent_events ORDER BY timestamp DESC LIMIT 20 Result: - Subagent tool calls (agent_id like 'a4f95a0b69d4fb1f4') DO have parent_event_id set (e.g. 'evt-25955942', 'evt-22d5ff8c') - UserQuery events have parent_event_id = None (correct) - Both main agent and subagent tool_calls share the SAME parent_event_id So agent_events DOES have some parent linking — but it links ALL tools from both parent AND subagent to the same UserQuery event, not Task->subagent tools. ### 3. pretooluse.py: parent_tool_use_id resolution logic is broken In create_start_event() (lines 535-555), the code tries to resolve parent_tool_use_id for tool_traces by looking up native_agent_id = tool_input.get("agent_id"). The problem: Claude Code does NOT pass agent_id in the PreToolUse hook input for tool calls made by subagents. The agent_id field in tool_input is not provided by Claude Code at PreToolUse time. So native_agent_id is always None, the lookup never runs, parent_tool_use_id stays None for every row. The suffix-stripping fallback (lines 486-503) relies on session_id ending in "-general-purpose", "-Explore" etc. — these suffix patterns may not match real subagent session IDs, so it also fails to find the parent task_delegation. ### 4. dashboard route: children are built from agent_events only (not tool_traces) The /views/activity-feed route calls activity_service.get_grouped_events(). That method uses agent_events.parent_event_id for hierarchy, NOT tool_traces.parent_tool_use_id. So tool_traces nesting is irrelevant to the dashboard view. ### 5. services.py get_grouped_events: TWO-LEVEL hierarchy only The dashboard builds hierarchy as: UserQuery (parent) -> children: direct children WHERE parent_event_id = uq_event_id Then separately (Step 3.5, lines 276-332) it attempts session-based re-parenting: - Finds Task events in children list - Extracts agent name from input_summary using regex r"\(([^)]+)\):" - Constructs sub_session_id = f"{uq_session_id}-{agent_name}" - Queries all events from that sub_session_id This Step 3.5 is the correct intent — fetch subagent events and nest them under their Task. BUT it fails because: a) The agent_id used in session construction may not match actual sub-session IDs b) The input_summary regex depends on a specific format like "(general-purpose): prompt" which may not always be present in the DB c) Even when sub-session events are found, they are added as task_evt["children"] — but the dashboard route (dashboard.py lines 240-255) only passes turn["children"] to the template as a FLAT list. It does NOT pass nested children-of-children. See dashboard.py line 241: "children": [c for c in children] — this is a flat list comprehension with no recursion. ### 6. template: only renders ONE level of nesting The template (activity-feed-hierarchical.html) renders: - group.parent = UserQuery row - group.children = flat list of direct children Children that themselves have children (task_delegation events with subagent tool calls nested under them) are NOT rendered — the template has no loop for grandchildren. --- ## Root Cause Summary (Priority Order) **Primary cause**: agent_events.parent_event_id is populated for subagent tool calls, but ALL point to the same UserQuery event_id rather than to their Task delegation event. So the hierarchy is UserQuery -> [all tools flat] instead of UserQuery -> Task -> [subagent tools]. **Contributing cause A**: dashboard.py flattens the data structure before passing to template — turn["children"] is built as a flat list comprehension (line 241-255), discarding any nested children returned by get_grouped_events recursive fetcher. **Contributing cause B**: The template only renders one depth level (parent + children), so even if data had task_delegation -> subagent_tools nesting, it would not render. **Contributing cause C**: parent_tool_use_id in tool_traces is always NULL because native_agent_id is never provided in PreToolUse hook input by Claude Code for subagent calls. This makes tool_traces unusable for hierarchy (though dashboard does not use it anyway). --- ## Exact Fix Needed ### Fix 1 (Data layer — most impactful) In posttooluse handler or event_tracker: when storing a tool_call event for a subagent, set parent_event_id to the task_delegation event_id (not UserQuery). This requires correctly identifying which task_delegation is the parent. The task_delegation event is already created in pretooluse.py create_task_parent_event(). The env var HTMLGRAPH_PARENT_EVENT is set to task_parent_event_id after Task() fires. But subagent tool calls run in a NEW PROCESS — env vars are not inherited from the parent Claude process. So HTMLGRAPH_PARENT_EVENT is not available to subagent hooks. The real solution: use agent_events WHERE event_type='task_delegation' AND status='started' and match by session_id (strip suffix from subagent session_id to get parent session_id, then find the most recent started task_delegation in that parent session). ### Fix 2 (Dashboard route — dashboard.py lines 240-255) Replace the flat list comprehension with a recursive function that preserves the nested structure from get_grouped_events, including children-of-children. ### Fix 3 (Template — activity-feed-hierarchical.html) Add rendering for deeper nesting levels (at least 3 deep: UserQuery -> Task -> tool calls). Currently only 2 levels rendered (parent + children). --- ## Files Involved - /Users/shakes/DevProjects/htmlgraph/src/python/htmlgraph/hooks/pretooluse.py - create_start_event() lines 532-555: parent_tool_use_id resolution broken - create_task_parent_event(): correct, creates task_delegation, sets env var - /Users/shakes/DevProjects/htmlgraph/src/python/htmlgraph/api/services.py - get_grouped_events() lines 36-263: Step 3.5 correct intent but fragile regex matching - children_sql line 104: WHERE parent_event_id = ? — correct query - /Users/shakes/DevProjects/htmlgraph/src/python/htmlgraph/api/routes/dashboard.py - activity_feed() lines 240-255: FLAT list comprehension loses nested structure - /Users/shakes/DevProjects/htmlgraph/src/python/htmlgraph/api/templates/partials/activity-feed-hierarchical.html - Only renders 2 levels deep (parent + children loop), no grandchildren

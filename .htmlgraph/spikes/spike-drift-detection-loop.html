Problem Statement

Repeated high-drift activities (0.71) detected while investigating the drift detection system itself:

  • Reading drift-queue.json to understand drift
  • Reading feature-self-tracking.html to get context
  • Activating htmlgraph-tracker skill to monitor the session

This creates a self-referential loop: actions taken to manage drift are themselves flagged as drift.

Questions to Answer

  • Should system self-monitoring activities be excluded from drift calculation?
  • Is the drift threshold (0.71) appropriate for this context?
  • Should drift-queue.json and feature HTML files be marked as "system" files?
  • How should we handle legitimate meta-work (understanding the system)?

Potential Solutions

  • Exclude `.htmlgraph/` directory files from drift scoring
  • Create an "investigation" activity type that doesn't trigger drift
  • Implement activity context tags (e.g., "system", "meta", "feature-work")
  • Use different drift thresholds for different activity types

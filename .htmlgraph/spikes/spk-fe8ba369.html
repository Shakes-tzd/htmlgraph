# HeadlessSpawner and Spawner Agent Integration Analysis ## Executive Summary HeadlessSpawner provides programmatic access to external AI CLIs while spawner agents serve as documentation guides. Critical gap: spawner agents document HOW to use HeadlessSpawner but are NOT automatically invoked. ## Key Findings ### 1. HeadlessSpawner Implementation (1065 lines) **Core Methods**: - spawn_gemini() - Gemini 2.0-Flash (FREE tier) - spawn_codex() - OpenAI Codex/GPT-4 (sandboxed) - spawn_copilot() - GitHub Copilot - spawn_claude() - Claude Code **Invocation**: Python API only, no CLI commands **Features**: Subprocess execution, HtmlGraph tracking, parent session context, event parsing ### 2. Spawner Agents (Documentation Scaffolds) **Files**: gemini-spawner.md, codex-spawner.md, copilot-spawner.md **Content**: Purpose, Workflow, Use Cases, Code Patterns, Error Handling, Fallback Strategy, Success Metrics **Role**: Document HOW to use HeadlessSpawner, not executable code ### 3. Integration Gaps 1. No automatic invocation (passive docs only) 2. No CLI commands (only Python API) 3. No centralized fallback logic 4. No cost estimation/comparison ### 4. Recommendations 1. Add CLI: htmlgraph spawn gemini/codex/copilot 2. Centralize fallback in HeadlessSpawner 3. Add cost estimation methods 4. Create auto-invocation hooks

Key Question
Why are tracker skill invocations and drift-queue reads receiving 0.71 drift scores when working on feature-self-tracking?

Investigation Steps

  1. Understand drift scoring algorithm - review how scores are calculated
  2. Analyze feature context patterns - what keywords/files should associate with feature-self-tracking
  3. Review recent high-drift activities - understand why Skill, Read, and feature management tools score as drift
  4. Identify classification gaps - determine if tool types need pattern refinement
  5. Document findings - create rules for improving drift detection accuracy

Problem Statement

The drift detection system is flagging activities as high-drift (0.71) when they are directly related to the active feature (feature-self-tracking). Specifically:

  • Reading drift-queue.json - infrastructure file for self-tracking feature
  • Reading feature-self-tracking.html - the active feature itself
  • Invoking htmlgraph:htmlgraph-tracker skill - directly supports feature tracking

These activities should have lower drift scores or be recognized as on-feature rather than off-feature.

Root Cause Hypothesis

The drift classifier may be using file path patterns or tool names that don't account for:

  1. Infrastructure files (.htmlgraph directory) that are meta to the feature itself
  2. Skill invocations that support feature coordination
  3. Feature management operations that are part of dogfooding the library

Success Criteria

After investigation:

  • Document the exact drift scoring algorithm
  • Identify which activities should NOT count as drift for feature-self-tracking
  • Propose specific pattern improvements for the classification system
  • Create test cases for edge cases discovered

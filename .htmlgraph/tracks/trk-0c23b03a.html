NetworkX Graph Engine Integration

Planned High Priority Created: 2026-03-15

Integrate NetworkX as the graph algorithm engine for HtmlGraph. HTML files remain the source of truth (git-friendly). Hyperlinks represent ANY relationship type — related_to, connected_to, blocked_by, depends_on, parent/child, references, authored_by, or any custom edge type. NetworkX provides battle-tested algorithms (shortest path, topological sort, cycle detection, PageRank, community detection, clustering) replacing hand-rolled BFS/DFS. The EdgeIndex feeds a nx.DiGraph powering all relational queries across ALL edge types equally.

Overview

Replace custom graph algorithms with NetworkX DiGraph, keeping HTML as storage. All relationship types are first-class — the engine is relationship-agnostic. blocked_by is just one edge type among many (related_to, connected_to, depends_on, parent/child, references, authored_by, custom).

Context

HtmlGraph's core philosophy is 'HTML is all you need' — HTML files are graph nodes, hyperlinks are edges, CSS selectors are the query language. Any relationship type is valid. The Edge.relationship field is already free-form and EdgeIndex indexes all types. However, graph.py has ~500 lines of hand-rolled BFS/DFS that often hardcode 'blocked_by' as the default. The graph_edges SQLite table is defined but never populated. Collections don't support relational queries. The CLI has no graph query commands. The integration must treat all relationship types equally.

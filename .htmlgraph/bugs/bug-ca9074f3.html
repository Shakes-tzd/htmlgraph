YOLO work-item hook blocks spawned subagents despite feature start being called in their session

Todo Medium Priority

Part Of:

Description

Observed 2026-04-10 during parallel dispatch of feat-bfa63ffa (sonnet) and feat-1a1de45b (opus) via Agent() with isolation=worktree. Both agents ran htmlgraph feature start as their first command (attribution worked, status changed to in-progress), but subsequent tool calls were blocked by the YOLO hook with: 'No active work item. Run: htmlgraph feature start ...' Both agents then went down rabbit holes investigating yolo_guard.go and permission_mode handling instead of doing their actual work. The sonnet agent burned 37k tokens, the opus agent burned 87k tokens — both producing zero useful output before being terminated. Root cause (hypothesis): the YOLO work-item guard uses per-session state that doesn't propagate from the parent to spawned subagent sessions. When an Agent() worktree subagent runs feature start, the CLI succeeds (updates HTML + DB), but the hook checking active work item at PreToolUse time looks in a different location (session file? env var? .active-session?) that was never set for the subagent's session ID. Evidence from the derailed agent transcripts: - 'launch_mode isn't used anywhere in the Go code. The YOLO check comes from event.PermissionMode or permission_mode in metadata' - 'but the error I got was No active work item...' - The agent found that feature start worked but the hook still blocked Reproduction: dispatch any subagent via Agent(isolation='worktree') into a YOLO-mode session and have it try to Edit a file after calling feature start. Expected: edit works. Actual: hook blocks with 'No active work item'. Impact: makes parallel agent dispatch unreliable in YOLO mode — the core workflow of /htmlgraph:execute is broken for anything non-trivial. Must be fixed before relying on parallel worktree execution. Fix direction: the work-item guard needs to resolve 'active work item' via HTMLGRAPH_PROJECT_DIR + .active-session (the canonical record), not via parent-session-scoped state. Or the spawned session needs to inherit the parent's active-session marker.

Observed 2026-04-10 during execution of trk-0677c709 launch readiness with 4 concurrent worktrees active. Full go test ./... runs produced two different test failures across two runs: Run 1: - TestGetOrSpawnHandshake — handshake timeout after 2s - TestGetOrSpawnConcurrent — spawn err: handshake timeout after 2s Run 2: - TestIdleReaperKillsStaleChild — child was not reaped after idle; children=1 In BOTH cases, go test ./internal/childproc/ in isolation passed cleanly (~0.7s to 1.4s). The failures reproduce only under parallel load from other worktrees running their own go test suites. Impact: pre-commit hook blocks docs-only commits because full suite fails. Forces retry loops and erodes confidence in the quality gate. CI runs concurrent jobs and may hit this. Suspects: - Hard-coded 2s handshake timeout in TestGetOrSpawnHandshake is too tight under CPU contention - IdleReaper timing assumption breaks when scheduler is backed up - Possibly a shared port/socket or filesystem resource that collides across parallel test runs Fix options: 1. Raise timeouts (cheap but masks a real timing bug) 2. Skip these tests when GOMAXPROCS-equivalent pressure is detected 3. Refactor tests to be deterministic — inject a clock instead of wallclock waits 4. Guard pre-commit hook to run childproc serially or with -p=1 Priority before launch since CI failures will undermine the deploy story.

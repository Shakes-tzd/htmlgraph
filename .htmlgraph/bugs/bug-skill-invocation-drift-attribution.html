Problem Statement

When skill invocations (e.g., Skill(htmlgraph:htmlgraph-tracker)) are used during feature development work, they are flagged as high-drift activities (0.71) instead of being correctly attributed to the active feature context.

Observed Behavior:

  • Multiple skill invocations during feature-self-tracking work are marked as independent high-drift activities
  • These activities are queued in drift-queue.json with 0.71 scores
  • They are not automatically associated with the parent feature
  • The drift detection treats tool overhead as a feature itself rather than a tool-use expense

Expected Behavior:

  • Tool invocations (Skills, reads, searches) should be attributed to the active feature context
  • Tool overhead shouldn't create independent high-drift work items
  • Multiple consecutive tool invocations should be grouped as a single activity unit
  • Drift score should reflect the work itself, not the tooling overhead

Root Cause Analysis

The drift detection algorithm is treating skill invocations as top-level activities rather than as overhead of the parent activity. Specifically:

  1. When a feature is being worked on, tool calls (Skills, Reads, etc.) are being tracked as independent activities
  2. These tool calls have inherent drift due to tool invocation overhead
  3. The algorithm doesn't have a concept of "child activities" or "tool overhead"
  4. Result: tool overhead gets flagged as drift instead of being attributed to the feature

Fix Steps

  1. Investigate drift detection algorithm in src/python/htmlgraph/
  2. Identify how activities are attributed to feature contexts
  3. Add concept of "activity grouping" or "parent context" to drift detector
  4. Modify drift calculation to exclude tool overhead from independent activity scoring
  5. Create test cases for repeated tool invocations within a feature context
  6. Verify that skill invocations during feature work are no longer flagged as high-drift
  7. Update drift-queue.json handling to re-classify on fix

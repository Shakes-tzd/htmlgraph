Missing CLI verb to revert a feature from in-progress back to todo

Todo Medium Priority

Part Of:

  • trk-0677c709

Description

Observed 2026-04-10. The htmlgraph feature CLI supports forward transitions (create → start → complete) and delete, but has no verb to revert a feature from in-progress back to todo when the work didn't actually happen. Concrete example: feat-1a1de45b (cross-project analytics). An opus agent ran htmlgraph feature start feat-1a1de45b in a worktree, then derailed before writing any code. The feature is now marked in-progress with zero commits behind it, polluting htmlgraph status 'In progress' and making the active work list misleading. Workarounds attempted: - htmlgraph feature update — only supports title/priority/track, not status - htmlgraph feature complete — would mark it done, which is a lie - htmlgraph feature delete — loses the description, edges, and history - Direct HTML edit — forbidden per project rules (all writes via Go CLI) - htmlgraph batch apply — create-only, not update Current best workaround: add-step with a DEFERRED note and leave it visibly in-progress. Unclean. Proposed fix: add 'htmlgraph feature revert ' or 'htmlgraph feature reset --to todo' that moves a feature back to todo, clears implemented_in edges from ghost sessions that produced nothing, and records a reason line. Same verb would apply to bug/spike/track. Priority: low/medium. Not a launch blocker but cleanup is needed for the status list to be trustworthy.

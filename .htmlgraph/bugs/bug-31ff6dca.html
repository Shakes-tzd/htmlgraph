Description

Observed 2026-04-10: drilling into the planbook project via the doorway shows 7 sessions with titles like 'why is my menubar not showing', 'is it possible for ghosty tabs to be a different color', 'a picture is a thousand words' — none of which are about planbook. Investigation confirmed the sessions ARE stored in planbook's htmlgraph.db with project_dir=/Users/shakes/DevProjects/planbook, so the per-project process isolation is working correctly at the DB boundary. The contamination is semantic, not architectural. Root cause: sessions are attributed to a project by CWD at SessionStart hook time. If a user is cd'd into planbook's directory but has a Claude Code conversation about unrelated topics (Ghostty, menubars, etc.), the session gets logged into planbook's .htmlgraph/htmlgraph.db. Later, when viewing planbook in the dashboard, those sessions surface as if they were planbook work. This pollutes the per-project sessions list and makes it harder for users to see what actually happened on a project. Options to discuss: 1. Filter the sessions list to show only sessions that touched files in the project, or that claimed at least one work item via feature/bug/spike start. 'Owned' sessions. 2. Add a two-tab split in the sessions view: 'Owned' (touched files or claimed work) and 'Visited' (session was active but unrelated to project work). 3. Require an explicit project scope signal beyond CWD — e.g., only attribute a session to a project if the session ran an htmlgraph CLI command against that project, or modified files in the project tree. 4. Drop CWD-based attribution entirely and attribute only via explicit signals (feature claims, CLI commands, file writes). Sessions with no signal stay unattributed. Needs design discussion before implementation — all four options have UX tradeoffs. Filed as todo under trk-cb36e595 because it surfaced during the hardening smoke test and shares the 'semantic isolation' concern even though the architectural isolation is solid.
